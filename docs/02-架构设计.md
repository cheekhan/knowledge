## 1. 架构总览

```
+--- 主进程 (Node.js) ---------------------------------------------------+
|  WindowManager      VaultService        chokidar Watcher               |
|  MenuManager        FileSystemService   MetadataService (sidecar)      |
|  IndexService (SQLite)   BindingService (绑定关系)   SettingsService    |
|  IPC 路由层: ipcMain.handle('namespace:action', handler)               |
+------------------------------------+-----------------------------------+
                                     | contextBridge (preload)
+------------------------------------+-----------------------------------+
| 渲染进程 (Chromium Sandbox)                                            |
|  React 18 + Zustand + react-resizable-panels                           |
|  FileTree(含绑定笔记) | NoteEditor(CM6) | PdfViewer(pdf.js) | Search   |
|  SplitLayout（左右分屏状态机）                                          |
+------------------------------------+-----------------------------------+
                                     |
+------------------------------------+-----------------------------------+
| 本地磁盘                                                                |
|  MyVault/                                                               |
|  ├── notes/*.md           ← 笔记（纯文件，唯一事实来源）                  |
|  ├── pdfs/*.pdf           ← PDF 原文件（只读，绝不修改）                  |
|  ├── pdfs/*.pdf.meta.json ← sidecar（标签/标注/进度/boundNotes）         |
|  ├── assets/              ← 笔记图片/附件                                |
|  └── .kb/                 ← index.db（FTS5 + bindings 冗余表）、app.json |
+-------------------------------------------------------------------------+
```

核心设计原则：

- **主进程管 IO，渲染进程管 UI**：文件读写、监听、SQLite 都在主进程；渲染进程不直接 touch `fs`
- **磁盘是唯一事实来源**：内存中的树、索引、绑定都是磁盘的投影；`index.db` 删掉后可全量重建
- **不修改原始文件**：PDF 原文件只读，标注/标签/绑定写入独立的 sidecar 文件
- **绑定即路径列表**：绑定关系的唯一事实来源是 sidecar `boundNotes`（库内相对路径数组），无需解析笔记内容
- **渲染引擎可替换**：`PdfEngine` 抽象接口，pdf.js 为默认实现

## 3. 进程边界与 IPC 契约

### 3.1 通信方式

| 方向 | 机制 | 用途 |
|------|------|------|
| 渲染 → 主 | `ipcRenderer.invoke` ↔ `ipcMain.handle` | 读写文件、搜索、绑定操作 |
| 主 → 渲染 | `webContents.send` → `ipcRenderer.on` | 文件变更、索引重建进度 |

### 3.2 IPC 通道定义（`shared/ipc-contract.ts`）

```typescript
export interface IpcApi {
  vault: {
    open(path: string): Promise<VaultSummary>;
    getTree(): Promise<FileTreeNode[]>;           // PDF 节点含 boundNotes 子节点

  asset: {

  };

  settings: {
    get(): Promise<AppSettings>;                       // 读 .kb/app.json

  window: {
    showInFolder(relPath: string): Promise<void>;       // shell.showItemInFolder
    revealVaultInOs(): Promise<void>;
  };

  search: {
    query(q: string, opts?: SearchOptions): Promise<SearchHit[]>;
    rebuildIndex(): Promise<void>;   // 进度经 vault 事件通道推送
  };
}

interface AppSettings {
  theme: 'system' | 'light' | 'dark';
  editorFontSize: number;
  autoSaveIntervalMs: number;          // 默认 1000
  lastLayout?: { splitRatio: number; pdfSide: 'left' | 'right' };

}

interface RecentVaultEntry { path: string; openedAt: string; }

interface BoundNoteInfo {

### 3.3 安全约束

- `contextIsolation: true`，`nodeIntegration: false`，`sandbox: true`
- 渲染进程只访问 `window.api`（白名单暴露）；所有 `relPath` 在主进程校验：拼接后必须在 Vault 根目录内，拒绝 `..` 与符号链接逃逸
- pdf.js 在渲染进程沙箱内运行，不授予额外权限

## 4. 主进程模块详设

### 4.1 VaultService（库管理）

- `open(path)`：校验可读 → 递归扫描 `.md`、`.pdf`、目录 → 生成 `FileTreeNode[]`（扫描时读取各 PDF 的 sidecar，把 `boundNotes` 嵌套为 PDF 节点的子节点）→ 创建/连接 `index.db` → 返回 `VaultSummary`
- 扫描性能：`fs.readdir` 递归，5000 文件 < 5s；后续由 chokidar 增量维护

### 4.2 FileSystemService（文件操作）

- **原子写**：写 `path.tmp` → `fsync` → `rename` 覆盖；崩溃只会留下 `.tmp`，不会截断原文件
- **回收站删除**：`shell.trashItem(path)`（Win/macOS 原生回收站）
- **重命名/移动**：先改磁盘 → 通知 BindingService 修复受影响 sidecar → 更新索引
- **删除笔记**：若为绑定笔记（存在于某 PDF 的 `boundNotes`），先从 sidecar 移除再删文件，避免悬空引用

### 4.3 FileWatcher（文件监听）

- chokidar：`ignoreInitial: true`，忽略 `.kb/` 与隐藏文件，事件防抖 100ms
- 事件归一化为 `VaultChangeEvent` → ①更新树缓存 ②增量更新索引 ③推送渲染进程
- 外部删除绑定笔记：Watcher 发现 `unlink` → 不主动改 sidecar（保留记录），`listBoundNotes` 返回时标记 `missing: true`，由用户在绑定面板一键清理。这样用户用 Git 同步、暂时性文件缺失（如移动硬盘未挂载）时绑定关系不会丢失
- 外部修改编辑中的笔记：无未保存更改 → 静默重载；有更改 → 弹冲突对话框（保留我的/加载磁盘/另存为）

### 4.4 MetadataService（sidecar 管理）

- 路径约定：`pdfs/abc.pdf` ↔ `pdfs/abc.pdf.meta.json`
- zod schema 校验（`PdfMetaSchema`）：`version/file/tags/lastPage/boundNotes/annotations`
- read-modify-write + 原子写；损坏时备份为 `.broken` 并返回空元数据，不阻塞阅读
- `lastPage` 防抖 2s 落盘；`tags/boundNotes/annotations` 变更立即落盘

### 4.5 BindingService（绑定关系，v2.0 新增核心模块）

**createBoundNote(pdfPath, opts)** 流程：

1. 确定目录：`opts.dir` 或默认 `notes/<PDF文件名去扩展名>/`（不存在则创建）
2. 确定文件名：`opts.name` 或 `《PDF名》笔记 N.md`（扫描 `boundNotes` 与目录取最大 N+1）
3. 生成初始内容：标题行 + 人读来源提示（如 `> 来源 PDF：pdfs/深入理解计算机系统.pdf`），**路径用正斜杠**（`path.posix.join` 保证跨平台一致）；该行纯人读，软件不作解析依据
4. 原子写 md 文件 → MetadataService 把路径 push 进 `boundNotes` → 更新 `bindings` 索引表 → 返回笔记相对路径
5. 渲染进程收到路径后进入分屏并打开编辑器

**路径修复（FR-6.5）**：

- 笔记 rename/move：遍历所有 sidecar，把 `boundNotes` 中的旧路径替换为新路径（只改匹配项，原子写）；同步更新 `bindings` 表
- PDF rename/move：PDF 自身 sidecar 跟随移动（`xxx.pdf.meta.json` 一起 rename）；`bindings` 表更新 `pdf_path`
- PDF 删除：其绑定笔记变为普通笔记（md 文件不动，sidecar 随 PDF 进回收站）；`bindings` 表清理对应行
- 全部修复操作聚合为事务性批次：先收集受影响 sidecar 清单 → 逐个原子写 → 失败项返回给 UI 提示

**查询**：

- `listBoundNotes(pdfPath)`：读 sidecar → 逐个 `fs.existsSync` 标记 missing → 取每篇 md 首个 H1 作为标题（有 mtime 缓存，避免每次全读）
- `pdfOfNote(notePath)`：查 `bindings` 索引表（O(1)），索引不可用时回退为全量扫 sidecar

### 4.6 IndexService（better-sqlite3 + FTS5）

```sql
CREATE TABLE notes (
  path TEXT PRIMARY KEY,
  title TEXT,
  mtime INTEGER
);

CREATE VIRTUAL TABLE notes_fts USING fts5(
  title, content,
  content='notes', content_rowid='rowid',
  tokenize='trigram'         -- 中文子串搜索，SQLite 3.34+ 内置
);

CREATE TABLE pdfs (
  path TEXT PRIMARY KEY,
  name TEXT,
  tags TEXT,                 -- JSON 数组冗余
  mtime INTEGER
);

-- P-13：PDF 文本内容索引（文本型 PDF 从流中提取 BT/ET 块文本）
CREATE VIRTUAL TABLE pdfs_fts USING fts5(
  path UNINDEXED,
  name,
  text_content,
  tokenize='trigram'
);

CREATE TABLE bindings (      -- 绑定关系冗余，事实来源是 sidecar
  pdf_path TEXT,
  note_path TEXT,
  PRIMARY KEY (pdf_path, note_path)
);
```

- better-sqlite3 同步 API，匹配 `ipcMain.handle` 模型
- 增量维护：Watcher 事件驱动，mtime 比对
- `bindings` 表由 BindingService 在每次绑定变更时同步更新；全量重建索引时从 sidecar 重新生成
- trigram 选型理由：内置零依赖，中文子串搜索可用；后续精度不足可升级 jieba

### 4.7 SettingsService（v2.0 补充）

- 读写 `.kb/app.json`（schema 由 `shared/types.ts` 的 `AppSettingsSchema`（zod）定义，**主/渲染共享同一 schema**，避免类型漂移）
- 默认值：`theme='system'`、`editorFontSize=15`、`autoSaveIntervalMs=1000`
- 最近库列表上限 10 条，按 `openedAt` 倒序；存储位置：用户目录 `app.getPath('userData')/recent-vaults.json`（不属于任何 Vault，跨库可见）
- `update(patch)`：read-modify-write + atomicWrite
- 启动时读 `lastVaultPath` 自动重开上次库（FR-1.2）

### 4.8 单实例锁（v2.0 补充）

```typescript
// main/index.ts
const gotLock = app.requestSingleInstanceLock();
if (!gotLock) {
  app.quit();   // 已有实例运行，退出
}
app.on('second-instance', () => {
  // 把已运行窗口聚焦，避免用户开第二份写同一 Vault 造成数据损坏
  if (mainWindow) {
    if (mainWindow.isMinimized()) mainWindow.restore();
    mainWindow.focus();
  }
});
```

理由：原子写（R5）只保证单进程内安全；多实例并发写同一 sidecar 仍会丢更新。单实例锁从根上消除该风险。

### 4.9 错误码规范（v2.0 补充，落地红线 R10）

所有主进程 service 抛出的错误统一为 `IpcError { code: string; message: string; detail?: unknown }`，`code` 取值约定：

| 前缀 | 含义 | 示例 |
|------|------|------|
| `E_PATH_*` | 路径安全/越权 | `E_PATH_OUTSIDE_VAULT`、`E_PATH_NOT_FOUND` |
| `E_FS_*` | 文件系统操作 | `E_FS_ATOMIC_WRITE_FAILED`、`E_FS_TARGET_EXISTS` |
| `E_META_*` | sidecar 元数据 | `E_META_SCHEMA_INVALID`、`E_META_BROKEN` |
| `E_BIND_*` | 绑定关系 | `E_BIND_PDF_NOT_FOUND`、`E_BIND_NOTE_EXISTS` |
| `E_INDEX_*` | 索引 | `E_INDEX_REBUILD_FAILED` |

IPC 层 try/catch 后 `reject(error)`；渲染层 service 统一 `catch` 转为 `Error` 重新抛出，组件层用 `code` 决定 UI 提示样式（toast vs 弹窗）。

### 5.1 状态管理（Zustand）

#### workspace.store.ts —— 分屏状态机（核心）

```typescript
type DockState =
  | { mode: 'note';  notePath: string }
  | { mode: 'pdf';   pdfPath: string; page: number }

状态变迁（v2.0 简化版，不再有"引用点击触发"路径）：

| 当前状态 | 触发动作 | 新状态 |
|----------|----------|--------|
| `note` | 文件树打开 PDF | `pdf` |
| `pdf` | 工具栏"创建笔记" | `split`（右侧为新绑定笔记） |
| `pdf` | 绑定面板/文件树点击绑定笔记 | `split`（右侧为该笔记） |
| `split` | 绑定面板切换另一篇绑定笔记 | `split`（仅换右侧 notePath，左侧 PDF 不动，阅读不中断） |
| `split` | 文件树打开普通笔记 | `note`（退出分屏） |
| `split` | 关闭任一侧 | 退化为 `note` 或 `pdf` |

布局状态（`splitRatio`、`pdfSide`）持久化到 `.kb/app.json`。

### 5.3 PdfViewer 关键设计

**PdfEngine 接口（可替换）**：

```typescript
interface PdfEngine {
  open(buffer: ArrayBuffer): Promise<void>;
  getPageCount(): number;
  getPage(n: number): Promise<{
    viewport: { width: number; height: number };
    renderTo(canvas: HTMLCanvasElement, scale: number): Promise<void>;
    getTextItems(): TextItem[] | null;   // 扫描版返回 null
  }>;
  close(): void;
}
```

**扫描版兼容**：AnnotationLayer 独立于 TextLayer 的 SVG overlay，文本型/扫描型同一代码路径。

**虚拟化渲染**：只渲染视口 ±2 页，canvas LRU 缓存上限 50 页，保证 300 页 PDF 内存 < 800MB。PageCanvas 通过 `renderedScaleRef` 跟踪上次渲染的缩放值，缩放变化时清缓存自动重新渲染，避免 `renderedRef` 阻止更新。侧栏通过 `sidePanelOpen` 状态控制显隐，PdfToolbar 提供 ◀/▶ 切换按钮。

**标注坐标**：sidecar 存归一化坐标（0~1），渲染时 × 当前页面像素尺寸；缩放/换屏不漂移。

### 5.4 NoteEditor 设计

- **源码模式**（默认）：CodeMirror 6 + Markdown 语法高亮，光标颜色 `.cm-cursor` 跟随 CSS 变量 `--text-primary`，选中背景 `.cm-selectionBackground` 使用 `--bg-active`
- **预览模式**：markdown-it 管线（GFM / KaTeX / Mermaid / highlight.js）
- 自动保存：变更防抖 1s → `note.write`；三态指示 `[保存中…] [已保存] [未保存]`；`beforeunload` 拦截
- 图片粘贴/拖拽：读 clipboard/拖拽文件 → `asset.save` → 光标处插入相对路径
- v2.0 删除：wikilink 自动补全、`[[pdf:…]]` 补全等引用类 CM6 扩展

### 5.5 Markdown 渲染管线

```typescript
const md = new MarkdownIt({ html: false, linkify: true })
  .use(gfmPlugin)          // 表格、任务列表
  .use(katexPlugin)        // $...$ / $$...$$
  .use(mermaidPlugin)      // ```mermaid → SVG
  .use(highlightPlugin);   // 代码块高亮
```

v2.0 起管线中**没有任何引用解析插件**，渲染即纯展示。

## 6. 关键机制时序

### 6.1 创建绑定笔记（核心路径）

```
用户：PDF 阅读中点击工具栏"创建笔记"
  │
  ▼ 渲染进程
弹出命名对话框（默认《PDF名》笔记 N）
  │ 确认
  ▼ ipc.invoke('binding:createBoundNote', pdfPath, { name })
主进程 BindingService：
  1. 定目录 notes/<PDF名>/（缺则建）
  2. 原子写 md（含 H1 标题 + 来源提示行）
  3. MetadataService：boundNotes.push(notePath)，原子写 sidecar
  4. IndexService：bindings 表 insert
  5. 返回 notePath
  │
  ▼ 渲染进程
workspace.store: pdf → split(pdfPath, page=当前页, notePath=新笔记)
  ├─ PdfViewer：保持现状（不重新加载 PDF，仅布局变化）
  └─ NoteEditor：加载新文件，聚焦，开始编辑
```

### 6.2 文件树同步

```
磁盘变更 → chokidar（防抖100ms）→ VaultService 更新树缓存
        → IndexService 增量更新 → send('vault:changed')
        → 渲染 vault.store 刷新 FileTree（含绑定子节点）
```

### 6.3 笔记重命名的绑定修复

```
note.rename(old, new)
  → FileSystemService 改磁盘
  → BindingService：扫全部 sidecar，boundNotes 中 old → new（原子写）
  → IndexService：notes 表 + bindings 表更新
  → 推送 vault:changed
```

### 6.4 PDF 引擎生命周期与分屏切换

**原则**：PDF engine 实例与 pdfPath 绑定，pdfPath 不变时 engine 不销毁重建。

**分屏切换场景**：从 `mode='pdf'` 切换到 `mode='split'` 时，React 因布局变化可能卸载 PdfViewer 组件。此时 engine 被 `useEffect` 清理函数销毁，新实例挂载后需检测 `engineRef.current` 是否为空来决定是否重建：

```typescript
// PdfViewer useEffect
if (pdfPath === pdf.pdfPath && engineRef.current) return  // engine 仍存活 → 跳过
// engine 为空 → 重新打开（即使 pdfPath 未变）
```

**额外保护**：`App.tsx` 中 `<PdfViewer key="pdf-viewer" />` 使用固定 key，防止 React reconciliation 因元素位置变化而错误卸载。

### 6.5 打开 PDF 恢复现场

```
打开 PDF → pdf.getMeta() 返回 { lastPage, tags, annotations, boundNotes }
  → PdfViewer 渲染并跳到 lastPage
  → binding.listBoundNotes() 标记 missing → BoundNotesPanel 展示
  → 若上次是 split 布局（.kb/app.json），自动恢复右侧笔记
```

## 9. 架构决策记录（ADR）

### ADR-001：Sidecar 元数据而非写入 PDF

不修改用户原始文件；JSON 读写远简单可靠；可进 Git、可迁移；删 sidecar 即回原始状态。

### ADR-002：SQLite 仅存索引，不存正文

避免双写不一致（用户直接改磁盘后 DB 过期）；索引随时可重建。

### ADR-003：FTS5 trigram 而非 jieba

内置零依赖，中文子串搜索够用；精度不足时后续可升级。

### ADR-004：Zustand 而非 Redux

无 boilerplate、TS 推导好、selector 精确重渲染。

### ADR-005：pdf.js 直用 + PdfEngine 接口抽象

react-pdf 封装是黑盒无法自由叠加 AnnotationLayer；接口抽象为 PDFium 后备留路。

### ADR-006：标注坐标归一化（0~1）

缩放无关、设备无关、窗口调整不漂移。

### ADR-007（v2.0 新增）：绑定关系存 sidecar 路径列表，而非解析笔记内容

- **背景**：v1.0 设计为"笔记中写 `[[pdf:x#page=1]]` 引用语法，解析全库笔记建立链接图"
- **决策**：改为 sidecar `boundNotes` 路径数组
- **理由**：
  - 引用解析需要全库扫描 + 链接修复（rename 时改写所有引用文件），复杂度高、易出错
  - 绑定关系在创建时即明确，路径列表 O(1) 可读，无需任何解析
  - 用户用外部编辑器写笔记时，不会意外"创建"或"破坏"关联
  - 一对多语义直接由数组表达，与需求完全对齐

### ADR-008（v2.0 新增）：外部删除绑定笔记时不主动清理 sidecar

- **决策**：Watcher 检测到绑定笔记被外部删除，只标记 `missing`，不修改 `boundNotes`
- **理由**：移动硬盘未挂载、Git 同步中途等暂时性缺失不应丢失绑定；用户在绑定面板手动"清理"才是明确意图

## 10. 风险与降级

| 风险 | 影响 | 缓解 |
|------|------|------|
| PDF.js 渲染复杂 PDF 失败 | 某文件无法阅读 | PdfEngine 接口预留 PDFium WASM 后备 |
| better-sqlite3 跨 Node 版本编译失败 | 安装/打包失败 | CI electron-rebuild 多平台矩阵 |
| sidecar 与 PDF 分离移动（用户只拷走 .pdf） | 标签/标注/绑定丢失 | 导出功能（P2）打包 pdf+sidecar；文档提示用户成对操作 |
| 大 Vault（>10K 文件）性能下降 | 启动慢 | 树虚拟滚动、异步初扫 + 后台渐进补全 |
| 绑定路径修复批量写 sidecar 中途失败 | 部分引用悬空 | 批次收集失败清单返回 UI；`listBoundNotes` 的 missing 机制兜底，用户可一键清理 |
