## Part A 公共规范与约束

以下规范**所有计划包必须遵守**，是包间能够独立、可拼接的前提。

### A.1 技术栈与版本基线

| 类别 | 选型 | 版本约束 |
|------|------|----------|
| 运行时 | Electron | ^43.x（内置 Node ≥ 24，Chromium ≥ 150） |
| 前端框架 | React | ^18.3 |
| 语言 | TypeScript | ^5.6，`strict: true` |
| 构建 | electron-vite | ^5.x（统一 main/preload/renderer 三目标） |
| 状态管理 | Zustand | ^4.x |
| 编辑器 | CodeMirror 6 | `@codemirror/*` 最新 6.x |
| PDF | pdfjs-dist | ^6.x（worker 通过 `import.meta.url` 打包） |
| 数据库 | better-sqlite3 | ^12.x（随 Electron 经 electron-rebuild 重编译） |
| 校验 | zod | ^3.x |
| 文件监听 | chokidar | ^3.x |
| Markdown | markdown-it | ^14.x + 插件（gfm 由 `markdown-it` 内置 enable 表格等） |
| 测试 | vitest | ^2.x |
| Lint | eslint + prettier | 使用 `@typescript-eslint` 推荐集 |

> 安装任何**未在此表出现**的新依赖前，必须在计划包交付说明中注明理由并确认许可证（仅允许 MIT/BSD/Apache-2.0）。

### A.2 代码规范

1. **TypeScript 严格模式**：禁止 `any`（确需时用 `unknown` 并收窄）；公共函数必须显式标注返回类型。
2. **命名**：
   - 文件：组件用 `PascalCase.tsx`，其余 `camelCase.ts`；服务类与文件同名（`BindingService.ts`）。
   - IPC 通道：`'namespace:action'`（如 `'binding:createBoundNote'`），命名空间 ∈ `{vault, note, pdf, binding, asset, search, settings, window}`。
3. **目录对齐**：新增代码必须落入 `02-架构设计.md` 第 2 章定义的目录结构；不允许在结构外新建顶层目录（`src/main`、`src/preload`、`src/shared`、`src/renderer` 之外的顶层 src 目录禁止）。
4. **注释**：公开接口写 TSDoc（`/** ... */` 含 `@param` `@returns`）；复杂算法写"为什么"而非"做了什么"。不加水印式注释、不留 TODO 占位到交付。
5. **提交**：每个计划包完成后 `pnpm lint && pnpm typecheck && pnpm test` 全绿；提交信息格式 `feat(P-XX): 描述`。

### A.3 架构约束（红线）

| # | 约束 | 说明 |
|---|------|------|
| R1 | **渲染进程禁止直接访问 Node API** | 一切文件/SQLite 操作必须经 `window.api`（contextBridge）。组件代码里出现 `import fs`、`import 'better-sqlite3'` 即违规。 |
| R2 | **组件不直接调 `window.api`** | 组件只能 import `src/renderer/services/*.service.ts`；service 是 window.api 的唯一封装出口。这样主进程接口变化只改一处。 |
| R3 | **共享类型只放 `src/shared/`** | `types.ts` / `ipc-contract.ts` 被 main 与 renderer 同时 import；shared 内禁止 import 任何 Node 或 DOM API（保持环境无关、纯类型 + 纯函数）。 |
| R4 | **PDF 原文件只读** | 任何代码不得以写模式打开 `.pdf`；标注/标签/绑定只能写 sidecar `*.pdf.meta.json`。 |
| R5 | **所有落盘写必须原子写** | 统一走 `FileSystemService.atomicWrite()`（tmp + fsync + rename）。禁止裸 `fs.writeFileSync` 写用户数据。 |
| R6 | **路径安全校验** | 主进程收到任何 `relPath`，必须经 `assertInsideVault()` 校验（拒绝 `..`、符号链接逃逸 Vault 根目录）。该校验函数在 P-01 提供，后续包必须复用。 |
| R7 | **绑定关系唯一事实来源是 sidecar `boundNotes`** | 禁止通过解析 md 内容推断绑定；SQLite `bindings` 表只是冗余加速，可被全量重建。 |
| R8 | **状态单向** | 渲染进程状态用 Zustand；跨组件共享状态必须放 store，禁止用 React 上下文层层透传业务数据。 |
| R9 | **坐标归一化** | 标注坐标一律 0~1 归一化存储，转换函数集中放 `renderer/lib/coord.ts`，禁止组件内各写一套换算。 |
| R10 | **错误处理** | 主进程 service 抛出的错误统一为 `{ code: string; message: string }`；IPC 层捕获后 reject，渲染层 service 统一 `try/catch` 转用户可读提示。 |

### A.4 契约先行规则（关键）

- **P-01（脚手架与公共契约）必须先完成并冻结**，因为它定义了 `shared/types.ts`、`shared/ipc-contract.ts`、preload 的 `window.api` 骨架、`constants.ts` 中所有常量（目录名、事件名、默认值）。
- P-02 ~ P-12 均**面向 P-01 的接口编程**：即使某个 IPC 尚未在另一端实现，也可以先按契约写代码 + mock，保证并行开发不阻塞。
- 任何计划包**需要新增/修改共享契约**时，先改 `shared/`，再改实现，并在交付说明中标注"契约变更"。

### A.5 环境约束

- 开发机：Node ≥ 20；macOS / Windows 均可开发；打包 macOS dmg 必须在 macOS 上，Windows NSIS 建议在 Windows 或 CI。
- 完全离线运行：禁止引入任何运行时网络请求（遥测、CDN）。pdf.js worker、字体、KaTeX/Mermaid 资源全部本地打包。

### P-01 工程脚手架与公共契约

**目标**：建立可启动的 Electron + React + TS 工程，冻结全部共享契约（类型、IPC 接口、常量），提供路径安全等基础工具，使后续所有包可面向契约并行开发。

**前置依赖**：无。

**涉及文件（新建）**：

```
package.json  tsconfig.json  electron.vite.config.ts  .eslintrc.cjs  .prettierrc
src/main/index.ts
src/main/window.ts
src/main/menu.ts
src/preload/index.ts
src/shared/types.ts
src/shared/ipc-contract.ts
src/shared/constants.ts
src/main/lib/vaultPath.ts        # assertInsideVault、toAbs、toRel
src/renderer/main.tsx
src/renderer/App.tsx             # 占位："等待 P-05 接入"
src/renderer/index.html
.vscode/settings.json（可选）
```

**实施步骤**：

1. 初始化 `package.json`：scripts 至少含 `dev`(electron-vite dev)、`build`(electron-vite build)、`start`、`lint`、`typecheck`(tsc --noEmit)、`test`(vitest run)。按 A.1 安装依赖（`better-sqlite3` 放 dependencies 并配置 `postinstall: electron-rebuild -f -w better-sqlite3`）。
2. `tsconfig.json`：`strict: true`、`target ES2022`、三处 include（main/preload 用 Node 类型，renderer 用 DOM 类型）——可用 `tsconfig.node.json` / `tsconfig.web.json` 分离。
3. `electron.vite.config.ts`：配置 main/preload/renderer 三入口；renderer 用 `@vitejs/plugin-react`；pdf.js worker 通过 `?worker` 支持。
4. `src/shared/constants.ts`：冻结常量——`SIDECAR_SUFFIX='.meta.json'`、`KB_DIR='.kb'`、`ASSETS_DIR='assets'`、`NOTES_DIR='notes'`、`PDFS_DIR='pdfs'`、`META_VERSION=2`、事件名 `'vault:changed'` 等、默认 `splitRatio=0.5`、缩放范围 `{min:0.5,max:4}`。
5. `src/shared/types.ts`：按需求 4.2 / 架构 3.2 定义——`PdfMeta`、`Annotation`、`NormalizedRect`、`FileTreeNode`（含 `boundNotes?: BoundNoteInfo[]`）、`BoundNoteInfo`、`VaultSummary`、`VaultChangeEvent`、`SearchHit`、`SearchOptions`、`IpcError`。**所有字段写 TSDoc**。
6. `src/shared/ipc-contract.ts`：完整 `IpcApi` 接口（架构 3.2 全文，含 `settings` / `window` 命名空间，**P-12 不再变更契约**）。此文件是 P-02~P-13 的唯一契约来源。
7. `src/main/lib/vaultPath.ts`：实现 `assertInsideVault(vaultRoot, relPath): string`（`path.resolve` 后以 `vaultRoot` 为前缀校验、解析符号链接后再比对）、`toAbs/toRel`。**这是红线 R6 的唯一实现**。
8. `src/main/index.ts` 启动时调用 `app.requestSingleInstanceLock()`，未获取锁则 `app.quit()`；`second-instance` 事件聚焦现有窗口（架构 4.8）。
9. `src/preload/index.ts`：用 `contextBridge.exposeInMainWorld('api', ...)` 按 `IpcApi` 形状暴露，方法体为 `ipcRenderer.invoke(channel, ...args)`；事件型方法（`onVaultChanged` 等）返回 `() => ipcRenderer.removeListener(...)` 作为 unsubscribe。**此时 invoke 的目标 handler 可不存在**（后续包实现），preload 先编译通过。
10. `src/main/window.ts` + `index.ts`：创建 BrowserWindow（`contextIsolation:true, nodeIntegration:false, sandbox:true`），加载 renderer；`menu.ts` 建最小原生菜单（macOS 必备 App 菜单）。
11. `src/renderer/main.tsx` + `App.tsx`：React 挂载，渲染占位文本。
12. 配置 eslint + prettier，保证 `npm run lint/typecheck` 通过。

**单元测试**（vitest，Node 环境）：

- `vaultPath.test.ts`：`assertInsideVault` 接受正常相对路径；拒绝 `../x`、`/abs/outside`、指向 Vault 外的符号链接。

**验收标准**：

- `pnpm dev` 能启动 Electron 窗口并显示占位 UI。
- `pnpm lint && pnpm typecheck && pnpm test` 全绿。
- `shared/` 两个契约文件齐全且被 preload import（证明可跨进程引用）。

**交付说明要求**：列出冻结的常量与 `IpcApi` 全部通道名（含 settings/window）；明确"契约已冻结，后续包如需变更须走 A.4 流程"。

### P-02 主进程：文件系统与监听

**目标**：实现库打开、文件树扫描、笔记 CRUD、原子写、回收站删除、chokidar 监听与变更推送。这是所有磁盘 IO 的地基。

**前置依赖**：P-01（契约、vaultPath 工具）。

**涉及文件**：

```
src/main/services/FileSystemService.ts
src/main/services/FileWatcher.ts
src/main/services/VaultService.ts
src/main/ipc/fs.ipc.ts            # 注册 note.* / vault.*
src/main/ipc/vault.ipc.ts
src/main/services/__tests__/FileSystemService.test.ts
src/main/services/__tests__/VaultService.test.ts
```

**实施步骤**：

1. **FileSystemService**：
   - `atomicWrite(absPath, content)`：写 `absPath + '.tmp'` → `fsync` → `rename`。
   - `readNote/writeNote/rename/move/delete`：`delete` 用 `shell.trashItem`；rename/move 先改磁盘再调用 `bindingRepairHook`（此处先留 `async (oldRel,newRel)=>{}` 空钩子，P-03 注入真实修复逻辑，避免 P-02 依赖 P-03 的实现）。
   - `createNote(parentDir,name,initialContent)`、`createFolder`。
   - 所有入参先 `assertInsideVault`。
2. **VaultService**：
   - `open(path)`：校验目录存在可读 → `scan()` 递归生成 `FileTreeNode[]`（`.md`/`.pdf`/目录；忽略 `.kb`、隐藏文件、`.tmp`）→ 缓存 `vaultRoot` 与树 → 调 `VaultService.readSidecarsIntoTree()`（此步在 P-03 前可返回不含绑定子节点的树；用 `try{ require MetadataService }catch{}` 或事件解耦，保持 P-02 可独立交付）。
   - `getTree()`、`getRecent()`（最近库列表存用户目录 `app.getPath('userData')/recent.json`，不属于 Vault）。
3. **FileWatcher**：
   - `watch(vaultRoot)`：chokidar `{ignoreInitial:true, ignored:[/(^|[\/\\])\../, '**/*.tmp']}`，100ms 防抖合并。
   - 归一化事件为 `VaultChangeEvent {type, relPath}` → 调 VaultService 局部刷新树 → `webContents.send('vault:changed', event)`。
   - 编辑中文件外部修改：维护 `openFiles: Map<relPath, dirtyFlag>`（dirtyFlag 由渲染层经 IPC 上报），未脏→推送重载，脏→推送冲突事件（UI 在 P-07 处理）。
4. **IPC 注册**（`fs.ipc.ts`/`vault.ipc.ts`）：把 `IpcApi.note`、`IpcApi.vault` 各方法接到 service；统一 `ipcMain.handle` + try/catch 包装为 `IpcError`。
5. 在 `main/index.ts` 中按顺序：`const fs = new FileSystemService(); const vault = new VaultService(fs); const watcher = new FileWatcher(...); registerFsIpc(...); registerVaultIpc(...)`。

**单元测试**：

- `FileSystemService.test.ts`：原子写后读回一致；模拟 rename 中断（写 tmp 不 rename）原文件不变；`delete` 调用 `shell.trashItem`（mock）；越界路径抛错。
- `VaultService.test.ts`：在临时目录造 notes/pdfs/assets 结构，断言树形状、忽略 `.kb`。

**验收标准**：

- 经 IPC 调用 `vault.open(示例库)` 返回正确树；`note.write` 后磁盘文件内容一致且过程出现 `.tmp`；在 Finder 中新建/删除文件，500ms 内收到 `vault:changed`。
- 红线 R5/R6 均有测试覆盖。

**交付说明要求**：标注"bindingRepairHook 为空钩子，待 P-03 注入"；列出 `vault:changed` 事件负载结构；说明 `window.showInFolder` handler 在本包实现（契约 P-01 已预留）。

### P-03 主进程：元数据与绑定

**目标**：实现 sidecar 读写（zod 校验）、PDF 元数据 IPC、BindingService 全部绑定逻辑（创建绑定笔记、解绑、查询、路径修复），并注入 P-02 预留的修复钩子。

**前置依赖**：P-01、P-02（FileSystemService.atomicWrite、修复钩子）。

**涉及文件**：

```
src/main/services/MetadataService.ts
src/main/services/BindingService.ts
src/main/ipc/pdf.ipc.ts
src/main/ipc/binding.ipc.ts
src/main/services/__tests__/MetadataService.test.ts
src/main/services/__tests__/BindingService.test.ts
```

**实施步骤**：

1. **MetadataService**：
   - `PdfMetaSchema`（zod）：`version/file/tags/lastPage/boundNotes/annotations/updatedAt`，字段默认值齐全。
   - `metaPathOf(pdfRel)`：换后缀 `.meta.json`。
   - `read(pdfRel)`：不存在→返回默认空 meta；JSON 解析或 zod 校验失败→重命名为 `.broken` 备份→返回空 meta。
   - `update(pdfRel, patch)`：read-modify-write + atomicWrite；`updatedAt` 自动刷新。
   - `updateLastPage(pdfRel, page)`：内部 2s 防抖合并后落盘。
2. **BindingService**：
   - `createBoundNote(pdfRel, {dir?, name?})`：按架构 4.5 五步——默认目录 `notes/<PDF去扩展名>/`、默认名 `《PDF名》笔记 N.md`（扫描现有 boundNotes 与目录取 N+1）、生成初始 md（`# 标题` + `> 来源 PDF：…` 人读提示行）、atomicWrite、push `boundNotes`、调用 `IndexService.upsertBinding`（用注入的回调，P-04 未就绪时为 no-op）、返回新笔记 relPath。
   - `unbind(pdfRel, noteRel)`：从 boundNotes 移除并落盘 + 通知索引删除。
   - `listBoundNotes(pdfRel)`：逐条 `existsSync` 标 `missing`；`title` 取 md 首个 H1（带 `{path,mtime,title}` 缓存，mtime 变才重读）。
   - `pdfOfNote(noteRel)`：优先查索引回调；索引不可用时全量扫 sidecar 回退。
   - `repairOnRename(oldRel,newRel,isDir)`：遍历 Vault 内全部 sidecar，替换 `boundNotes` 中匹配项（dir 则前缀替换），逐个 atomicWrite，收集失败清单返回。
   - `repairOnPdfRename/delete`：sidecar 随 PDF rename；PDF 删除时 bindings 清理。
3. **注入 P-02 钩子**：在 `main/index.ts` 组装处，`fsService.setRenameHook(binding.repairOnRename)`，替换 P-02 的空钩子。
4. **IPC**：实现 `IpcApi.pdf`（`readBuffer` 用 `fs.readFile` 返回 Buffer→ArrayBuffer）、`IpcApi.binding` 全部方法。

### P-13 P2 增强功能

**目标**：实现一期标记为 P2 但仍需交付的三项功能：示例库、PDF 页内文字搜索、PDF 文本索引搜索。这三项彼此独立，可拆给不同执行者并行。

**前置依赖**：P-12（基础能力就绪）。

**涉及文件**：

```
# 示例库（FR-1.3）
resources/default-vault/notes/欢迎.md
resources/default-vault/pdfs/example.pdf
resources/default-vault/pdfs/example.pdf.meta.json   # 预置 1 标签 + 1 标注 + 1 绑定笔记
resources/default-vault/pdfs/example/《example》笔记 1.md
src/main/services/DefaultVaultService.ts              # 首次启动拷贝到 userData/default-vault

# PDF 页内搜索（FR-4.5）
src/renderer/components/pdf/PdfFindBar.tsx
src/renderer/components/pdf/engine/pdfFindController.ts  # 包装 pdf.js findController

# PDF 文本搜索（FR-7.3）
src/main/services/PdfTextExtractor.ts                # pdf.js getTextContent 提取，写 pdfs_fts
src/main/ipc/search.ipc.ts（扩展 query 支持 scope='pdfs'）
```

**实施步骤**：

1. **示例库**：在 `resources/default-vault/` 内放预制的欢迎笔记 + 小体积带文本层 PDF + 对应 sidecar（含标签、标注、绑定笔记）；首次启动检测 `recent-vaults.json` 为空且 `userData/default-vault/` 不存在 → 拷贝 → 自动打开。
2. **PDF 页内搜索**（FR-4.5）：PdfToolbar 加"查找"按钮 → 弹 PdfFindBar（输入框 + 上/下一条 + 高亮计数）；调 pdf.js `PDFPageProxy.getOperatorList` 或全局 `eventBus` 的 `find` 事件驱动 `findController`；仅文本型 PDF 可用，扫描版按钮禁用并提示。
3. **PDF 文本搜索**（FR-7.3）：IndexService 扩展——索引 PDF 时调 `PdfTextExtractor.extract(pdfPath)`（pdf.js 主进程可跑，但需 `pdfjs-dist/legacy` 或单独 worker）写入 `pdfs_fts` 表；查询时 `scope='pdfs'` 走该表；扫描版无文本跳过并标记。

**单元测试**：

- DefaultVaultService：首次拷贝正确，二次启动不覆盖。
- PdfTextExtractor：文本型 PDF 提取非空，扫描版返回空数组。

**验收标准**：示例库一键可玩；文本型 PDF 页内搜索高亮正确；全局搜索能命中 PDF 内文本。

**交付说明要求**：说明 PDF 主进程文本提取的 worker 方案（避免阻塞主进程）。
