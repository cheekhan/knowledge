## Part A 公共规范与约束

以下规范**所有优化包必须遵守**，与 `03-开发计划.md` Part A 一致；优化特有约束另行列出。

### A.1 继承的架构红线（不可违反）

| # | 约束 | 说明 |
|---|------|------|
| R1 | 渲染进程禁止直接访问 Node API | 文件读取仍经 `window.api` |
| R2 | 组件不直接调 `window.api` | 经 `src/renderer/services/*.service.ts` |
| R3 | 共享类型只放 `src/shared/` | 契约变更须先改 shared |
| R4 | PDF 原文件只读 | 书签写入 sidecar，禁止写 PDF 二进制 |
| R5 | 落盘写必须原子写 | `FileSystemService.atomicWrite()` |
| R8 | 状态用 Zustand | 书签列表可放 pdf.store，不新增全局 Context |

### A.2 优化特有约束

1. **不引入新的原生 C++ 模块**（与 ADR 及 `02-架构设计.md` §7.5 一致）。
2. **性能优化不得破坏已有功能**：虚拟化、缩放重渲染、lastPage 恢复、分屏不 reload、标注侧栏、页内搜索（FR-4.5）在 O-01 完成后仍须可用。
3. **书签纠正为需求变更**：删除「文件级标签（tags）」概念，替换为「PDF 书签（bookmarks）」；须做 sidecar schema 迁移，不得静默丢数据（见 O-02 §迁移策略）。
4. **每个优化包交付前**：`pnpm lint && pnpm typecheck && pnpm test` 全绿。
5. **禁止仅改 UI 文案**：「标签」→「书签」须同步改数据模型、索引、文档，不是 rename label。
6. **实测数据要求**：O-01 交付说明须附至少一组性能对比（见 O-01 §验收）。

### A.3 推荐执行顺序

```
O-03（环境与基线同步，低风险，1 天内）
  ↓
O-01（PDF 性能，改动 PdfViewer 核心，2~4 天）
  ↓
O-02（书签功能，改动 side panel + schema，1~2 天）
  ↓
Part D 文档同步 + Part E 总验收
```

> O-01 与 O-02 均修改 `PdfViewer.tsx`，**必须串行**：先 O-01 稳定渲染管线，再 O-02 改侧栏。O-03 可与 O-01 并行，但涉及 pdfjs worker 配置的项应在 O-01 开始前完成。

### O-01 PDF 预览性能优化

**目标**：打开大体积、多页 PDF 时，**2 秒内出现首屏内容**（toolbar + 第 1 页或目标页骨架/低清图），连续滚动 300 页内存 < 800MB，无明显掉帧。

**前置依赖**：无（建议 O-03 中 pdfjs worker 配置项先完成）。

#### 涉及文件

```

src/renderer/components/pdf/engine/PdfJsEngine.ts  # 懒尺寸、outline 页码解析（与 O-02 共享）
src/renderer/components/pdf/engine/PdfEngine.ts    # 接口扩展
src/renderer/components/pdf/useVirtualPages.ts     # 重写为 scrollTop 驱动
src/renderer/components/pdf/pageCache.ts           # 可选：渲染任务取消令牌
src/renderer/components/pdf/renderQueue.ts         # 新增：渲染优先级队列
src/renderer/components/pdf/usePageDimensions.ts   # 新增：懒加载尺寸 hook

electron.vite.config.ts                            # pdfjs 分包 / worker 优化（若 O-03 未做）
```

#### 分步实施

##### O-01.1 分阶段加载 — 消除白屏

**现状**：`pdf.loading` 为 true 时 `return <div>加载中…</div>`，用户看不到任何 PDF UI。

**改造**：

1. 将打开流程拆为三阶段：
   - **Stage A（阻塞最短）**：`readBuffer` → `engine.open` → 得到 `pageCount` → **立即** `setPageCount` + `pdf.open(...)` + `setLoading(false)`，渲染 toolbar + 空白滚动区。
   - **Stage B（后台）**：并行 `getMeta` + `listBoundNotes`；完成后更新 store，不阻塞 Stage A。
   - **Stage C（后台）**：懒加载页面尺寸（见 O-01.2）；首屏仅解析目标页 ±1 的尺寸。

2. `loading` 语义变更：
   - `loading`：仅表示 engine 尚未 open（极短）。
   - 新增 `dimensionsReady: boolean` 或 `readyPages: Set<number>`：尺寸未就绪的页显示 **灰色骨架占位**（高度用估算值，见 O-01.2），而非全页白屏。

3. lastPage 跳转：
   - Stage A 结束后即 `setPage(targetPage)` + `userJumpRef.current = true`。
   - 目标页尺寸未就绪时用估算高度 scroll（O-01.2），尺寸到达后 **不跳动**（一次修正 scrollTop）。

**验收**：打开 300 页测试 PDF，从点击到看见 toolbar ≤ 500ms（本地 SSD）；1 秒内看见目标页骨架或内容。

##### O-01.2 懒加载页面尺寸 — 移除全量 getPageDimensions

**现状**：`PdfJsEngine.getPageDimensions()` 循环 `1..numPages` 调用 `getPage(i)`。

**改造**：

1. **删除**打开流程中对全量 `getPageDimensions()` 的 await。

2. 新增 `usePageDimensions(engine, pageCount)`：
   ```typescript
   // 伪代码契约
   interface PageDimensionsState {
     /** 已解析的真实尺寸；未解析的页为 null */
     dims: (PageDimension | null)[]
     /** 获取单页尺寸（带 memo + 并发上限） */
     ensureDim(pageNum: number): Promise<PageDimension>
     /** 估算高度（用于未加载页）：默认 A4 ratio 或第 1 页尺寸 */
     estimateHeight(pageNum: number, scale: number): number
   }
   ```

3. `PdfJsEngine` 新增：
   ```typescript
   getPageDimension(num: number): Promise<PageDimension>  // 单页，内部 getPage + getViewport(1)
   ```
   保留 `getPageDimensions()` 供测试/重建索引用，**禁止**打开路径调用。

4. 加载策略：
   - 优先队列：`targetPage`, `targetPage±1`, 可见范围, 其余。
   - 并发上限 `PDF_DIM_FETCH_CONCURRENCY = 4`（写入 `constants.ts`）。
   - 用 `Map<number, Promise>` 去重，避免同页重复请求。

5. 总高度 `totalHeight` 计算改为：
   ```typescript
   sum = dims.reduce((s, d, i) => s + (d ? ceil(d.h*scale) : estimateHeight(i+1, scale)) + gap, 0)
   ```
   随 dims 填充逐步精确，滚动条长度平滑变化（允许轻微抖动，但 scroll 位置应稳定）。

**验收**：500 页 PDF 打开时不发起 500 次顺序 getPage；Chrome Performance 面板可见尺寸请求分批、可取消。

##### O-01.3 DOM 窗口化 — 仅挂载可见页 DOM

**现状**：`Array.from({ length: pageCount })` 创建全部 page wrapper。

**改造**：

1. 重写 `useVirtualPages` → `useScrollWindow`：
   - 输入：`scrollTop`, `viewportHeight`, `pageCount`, `getPageTop(pageNum)`, `getPageHeight(pageNum)`
   - 输出：`visibleRange: { start, end }`（**纯数学计算**，不依赖 page DOM）
   - buffer 默认 2（沿用 `PDF_VISIBLE_BUFFER = 2`）

2. 渲染层 **只** `map` `visibleRange.start..end` 的页码创建 DOM：
   ```tsx
   {rangePages.map(pageNum => (
     <div key={pageNum} style={{ position:'absolute', top: getOffsetTop(pageNum), ... }}>
       <PageCanvas ... />
     </div>
   ))}
   ```

3. 滚动容器内保留 **单一** 高度占位 `<div style={{ height: totalHeight }} />`（无子 page div）。

4. `onScroll` 页码计算（L209-233）改为基于 `scrollTop + pageDimensions/estimate`，移除 `getBoundingClientRect` 遍历。

5. `jumpToPage` / `scrollIntoView` 改为：
   ```typescript
   scrollRef.current.scrollTop = getOffsetTop(pageNum)
   ```

**验收**：300 页 PDF 运行时 DOM 中 page wrapper ≤ 7 个（±2 buffer + 当前）；Elements 面板可验证。

##### O-01.4 渲染优先级队列

**现状**：每个 `PageCanvas` 独立 `useEffect` 触发 render，无优先级。

**改造**：

1. 新增 `src/renderer/components/pdf/renderQueue.ts`：
   ```typescript
   class PdfRenderQueue {
     enqueue(pageNum: number, priority: number, task: () => Promise<void>): void
     cancel(pageNum: number): void
     clear(): void
   }
   ```
   - priority：0 = 当前页，1 = ±1 页，2 = 可见缓冲，3 = 其他
   - 同页新任务覆盖旧任务；组件 unmount 时 cancel

2. `PageCanvas` 渲染经 queue 调度，不再直接 `engine.getPage().then(render)`。

3. 缩放变化：`queue.clear()` + `cache.clear()` + 按新 scale 重入队可见页。

4. 可选优化：滚动中（`isScrolling` debounce 150ms）用 `scale * 0.75` 低清快速渲染，停止滚动后全清度渲染（须可配置关闭，默认开启）。

**验收**：快速滚到第 200 页，当前视口页优先出现内容，无明显「先渲远处页」现象。

##### O-01.5 大文件 IPC 优化（进阶，建议实施）

**现状**：整文件读入 ArrayBuffer。

**改造（Range 加载）**：

1. 主进程新增 IPC `pdf:openHandle(relPath) → { fileSize, cacheKey }`（或复用 readBuffer 分阶段）。

2. 渲染进程 `pdfjs-dist` 使用自定义 `PDFDataRangeTransport` / `getDocument({ url, range })`：
   - 通过 IPC `pdf:readRange(relPath, begin, end)` 按 64KB~1MB chunk 读取
   - pdfjs 按需拉取，**不全量加载**

3. `PdfJsEngine.open` 签名扩展：
   ```typescript
   open(source: ArrayBuffer | { relPath: string }): Promise<void>
   ```

4. 小文件（< 10MB）仍走 ArrayBuffer 快路径，避免 IPC 往返开销。

**若本步工期不足**：在交付说明中标注为「Phase 2」，但 O-01.1~O-01.4 必须完成。

**验收**：打开 80MB+ PDF，渲染进程堆内存峰值明显低于「文件大小 × 2」。

##### O-01.6 Canvas 渲染质量

1. `renderTo` 中考虑 `devicePixelRatio`（上限 2），避免高 DPI 屏模糊或过载：
   ```typescript
   const outputScale = Math.min(window.devicePixelRatio || 1, 2)
   canvas.width = floor(svp.width * outputScale)
   // viewport scale 相应调整
   ```

2. 单页 canvas 像素上限：若 `width*height > PDF_MAX_CANVAS_PIXELS`（如 16MP），自动降低 outputScale 并打 debug 日志。

### O-02 PDF 书签功能纠正

**目标**：删除错误的「文件级标签（tags）」实现，改为正确的 **PDF 书签**：展示 PDF 内置目录（Outline）、支持用户添加/删除/跳转书签；侧栏与全局面板 UI 语义统一为「书签」。

**前置依赖**：O-01 完成（PdfViewer 渲染管线稳定）。

#### 概念定义（执行 agent 必须遵守）

| 概念 | 含义 | 存储位置 |
|------|------|----------|
| **PDF 内置目录（Outline）** | PDF 文件内嵌的书签/大纲树 | 只读，来自 PDF；`PdfJsEngine.getOutline()` |
| **用户书签（User Bookmark）** | 用户手动添加的页级导航点 `{ title, page }` | sidecar `bookmarks: []` |
| ~~文件级标签（tags）~~ | ~~整份 PDF 的分类词~~ | **删除**，不再使用 |

> 「PDF 标签功能就是书签功能」= 产品要的始终是 **页级导航书签**，不是 Obsidian 式的文件标签。

   // 兼容旧 sidecar：version 2 且存在 tags 且无 bookmarks

##### O-02.2 修复 PDF 内置目录解析

**现状**：`PdfJsEngine.conv()` 把 `dest` 当 number，实际 pdfjs 返回的是 `string | Array | null` 引用。

**改造** `getOutline()`：

```typescript
async getOutline(): Promise<OutlineItem[]> {
  const outline = await this.doc.getOutline()
  if (!outline) return []
  return Promise.all(outline.map(item => this.resolveOutlineItem(item)))
}

private async resolveOutlineItem(item: pdfjsLib.OutlineNode): Promise<OutlineItem> {
  let page = 1
  if (item.dest) {
    let dest = item.dest
    if (typeof dest === 'string') {
      dest = await this.doc!.getDestination(dest)
    }
    if (Array.isArray(dest)) {
      const pageRef = dest[0]
      page = (await this.doc!.getPageIndex(pageRef)) + 1  // 0-based → 1-based
    }
  }
  const children = item.items?.length
    ? await Promise.all(item.items.map(c => this.resolveOutlineItem(c)))
    : undefined
  return { title: item.title || '未命名', page, children }
}
```

**`OutlineItem` 扩展**（`PdfEngine.ts`）：
```typescript
export interface OutlineItem {
  title: string
  page: number
  children?: OutlineItem[]
}
```
